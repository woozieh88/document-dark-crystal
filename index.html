0. Production Framing

ProductDark Crystal — PvP Map
ClientDAIGON ESPORTS
Studio delivery targetMinecraft Education Edition
AudienceStudents aged 8–15
Session size8–16 players
Match typeConversion / Last Survivor
Core delivery expectationone themed arena, one repeatable session loop, one command/add-on implementation layer, and one telemetry-ready scoreboard/event baseline.

Builder-wide standard

Builders must treat the world as a competitive arena first and a scenic environment second. Every decorative decision must preserve route readability, player silhouette visibility, and fair traversal. The world must feel premium, branded, and dramatic, but never visually noisy or mechanically confusing.

Developer-wide standard

Developers must implement the round loop, conversion system, ready flow, scoreboard outputs, reset flow, and event triggers as deterministic state transitions. The map must never depend on manual cleanup between rounds. Every round must be startable, completable, and resettable without teacher intervention unless teacher override is explicitly invoked.

1. Overview

Core premise

“Dark Crystal is a PvP conversion map for 8–16 students aged 8–15. One player begins as the Crystal Bearer — they carry a corrupted dark crystal and must touch other players to spread its power. Every player they convert also becomes a Bearer. The last uncorrupted player wins the round. The map auto-resets and a new Bearer is randomly selected each round.”

Alternative Note: This overview defines conversion by touch, while later sections define conversion by weapon hit / damage event. One authoritative trigger must be chosen before implementation lock. If command-block reliability is prioritized, proximity/touch conversion is lower risk than true PvP-hit detection.

Builder implementation

  • The map must support continuous pursuit, line-of-sight breaks, and repeated route decisions.
  • Spatial pacing must create three emotional states: scatter, chase, and final-survivor tension.
  • No zone may be so isolated that a single survivor can stall the round indefinitely.
  • Arena composition must visually communicate spreading corruption from a central origin.

Developer implementation

  • Implement a round-state machine with at minimum: hub_idle, ready_check, countdown, round_live, round_end, reset.
  • Maintain explicit player role state: neutral, survivor, bearer, optionally spectating.
  • Auto-reset must clear role tags, loadouts, timers, and transient FX before the next ready cycle.
  • Random Bearer selection can use Bedrock target selectors and scoreboards; @r is documented as selecting one random living player.

Role fantasy

“The Crystal Bearer is powerful and cool — this is a fantasy adventure mechanic. Bearers should feel like a powerful force in an ancient magical world. The map is designed so every student experiences both sides across a session: the thrill of being the pursuer and the tension of being the last survivor.”

Builder implementation

  • The arena must make Bearers readable at distance through open lanes, elevated sightlines, and deliberate reveal moments.
  • The environment must support fast chase lines for Bearers and layered escape choices for Survivors.
  • Visual composition must frame Bearers as dominant through crystal lighting, route prominence, and stage-like encounter spaces.

Developer implementation

  • Bearers need a persistent mechanical advantage that is felt immediately: movement advantage, clearer target acquisition, stronger visual identity, and fast conversion loop.
  • Session design must rotate roles naturally across rounds through random assignment, not manual balancing.
  • Round count and timing must be tuned so multiple students plausibly experience both roles during one session.

Arena identity

“The map should feel like a DAIGON Esports arena: branded throughout, with dramatic moments, a live scoreboard, and mechanics that make every session produce memories students carry out of the room.”

Builder implementation

  • Branding must be architecturally embedded, not pasted on as afterthought signage.
  • The Spawn Hub and Town Square are primary spectacle zones and must carry the strongest branded presentation.
  • Major reveal shots must include the Crystal pedestal, leaderboard, and DAIGON insignia.

Developer implementation

  • “Dramatic moments” must be surfaced via titles, timed audio, fireworks, visible state changes, and escalating feedback.
  • The scoreboard must be readable during live play and useful between rounds.
  • Major session beats must be broadcast cleanly to all players.

2. Core Specifications

Players: “8 to 16”

Builder implementation

  • Routes, hub capacity, and confrontation spaces must be sized for 16 bodies without collision-heavy crowding.
  • Town Square, gate mouth, rooftop chokepoints, and hub wall spacing must be tested at full occupancy.

Developer implementation

  • All selectors, scoreboards, win logic, and ready checks must support 16 concurrent players without relying on fixed player names.
  • All state logic must be dynamic and count active players at runtime. Bedrock target selectors and scoreboards support this style of implementation.

Teams: “None — free for all. Crystal Bearers vs Survivors.”

Builder implementation

  • Avoid red-vs-blue spatial assumptions.
  • The map must read clearly without team-owned territories.

Developer implementation

  • Use role tags/objectives rather than formal competitive teams.
  • Roles are temporary session states, not lobby teams.

Match Format: “Last survivor wins the round. 3–4 rounds per session.”

Builder implementation

  • Match flow must reset quickly enough that 3–4 rounds fit comfortably into the intended classroom session.
  • Round-end spectacle must be short and repeatable.

Developer implementation

  • Session controller must track round count and stop at configured total rounds.
  • Session-level winner logic must aggregate round results.

Round Length: “3–5 minutes per round (teacher-configurable)”

Builder implementation

  • Arena size, route density, and rooftop access must still produce contact within 3 minutes.
  • There must be no overlarge hiding networks.

Developer implementation

  • Expose timer mode selection through teacher-facing NPC or equivalent control.
  • Scoreboard timer can be implemented through command-block/scoreboard logic. Scoreboards are documented for use with commands and command blocks, and sidebar display is supported.

Win Condition: “Last uncorrupted survivor, OR player with most rounds survived at session end”

Builder implementation

  • End-of-round presentation zones must bring attention back to the Spawn Hub / Crystal pedestal.

Developer implementation

  • Distinguish round_winner from session_winner.
  • Session winner = highest rounds won at session end.
  • Draw handling must not corrupt session winner calculations.

Capture Object: “The Dark Crystal — glowing End Crystal on a branded obsidian pedestal”

Builder implementation

  • Treat as primary world landmark and thematic origin point.
  • Must be visible, protected from player interference, and framed cinematically.

Developer implementation

  • The object is thematic, not a pick-up.
  • It acts as a visual anchor for titles, fireworks, and branded effects.

Platform: “Minecraft Education Edition”

Builder implementation

  • Avoid reliance on retail-only content assumptions unless validated in Education.
  • Block palette, entity usage, and pack structure must remain compatible with Education deployment expectations.

Developer implementation

  • Prefer stable Bedrock-compatible command patterns, command blocks, scoreboards, NPC interaction, and add-on structures that align with documented creator workflows. Minecraft commands, scoreboards, command blocks, NPC dialogue, add-ons, and scripting are all documented in the current creator stack.

Ready System / Spawn System / Map Theme

Builder implementation

  • Spawn Hub must cleanly support waiting, briefing, scoreboard reading, and round dispatch.
  • Theme language is fixed: ancient ruined town, warm stone, overgrowth, crystal accents, lantern light.

Developer implementation

  • Ready gating must be hard-blocked until all required players are ready or teacher override fires.
  • Gate state must synchronize with round state.

3. The Dark Crystal

Crystal centerpiece

“The Dark Crystal is the centrepiece of the map — a glowing End Crystal mounted on a polished obsidian and blackstone pedestal at the heart of the Spawn Hub. It is the origin of the Bearer power and the most visually striking element in the arena.”

Builder implementation

  • Pedestal must sit on the main visual axis of the hub.
  • Surround with clear walkable circulation so players can gather without obstructing view.
  • Use blackstone/obsidian massing with purple glass and controlled uplighting.
  • This asset must feel ceremonial, not decorative filler.

Developer implementation

  • Use the pedestal as origin for ceremony events: round-start sound, victory fireworks, session-end fireworks, potentially camera facing.
  • If using a fixed overhead spectator/camera shot, align the world composition around this landmark.

3.1 The Bearer Mechanic

Bearer selection

“At round start, one player is randomly selected as the Crystal Bearer. The selection is announced via a /title screen and a sound effect.”

Developer implementation

  • Use deterministic round-start chain: pick candidate → clear prior states → assign bearer role → broadcast messaging → apply kit/effects → open countdown.
  • Random target selection is supported by documented selectors such as @r.

Bearer loadout

“The Bearer receives:

  • Speed II for the duration of the round
  • A renamed Sword — ‘Crystal Blade’ — with Sharpness I and a purple enchantment glow
  • A purple particle aura pulsing from the Bearer every tick
  • A custom tag above their head: ‘BEARER’”

Alternative Note: /tag stores internal entity tags, but it does not create a visible overhead label. If “BEARER” must appear above the player’s head, use a visible nametag approach or a below-name scoreboard display instead of relying on /tag alone. Official docs distinguish entity tags from visible name-tag behavior, and below-name is a supported scoreboard display slot.

Builder implementation

  • Bearer visibility must be supported by lane width, skyline exposure, and rooftop sightlines.
  • Particle readability must not be lost against purple-heavy architecture; keep aura contrast high.

Developer implementation

  • Speed II is a persistent round buff.
  • The weapon must be granted on assignment and removed on reset.
  • The purple aura can be done with repeated /particle commands or custom particle content in a resource pack. Creator docs support particle effects and custom particle content.
  • A renamed Crystal Blade may be easiest as a custom item/add-on asset or via supported item component workflows rather than a bare vanilla sword assumption.

Conversion flow

“When the Bearer hits a Survivor, that Survivor is immediately converted — they receive the same speed boost, weapon, particle effect, and BEARER tag. Conversion triggers a brief flash and a boom sound so the whole map knows it happened.”

Alternative Note: Reliable player-on-player hit detection is a high-risk requirement if the baseline is command blocks only. Bedrock command documentation supports selectors, scoreboards, and damage commands, but does not document a simple command-block event hook for “player A damaged player B.” If the project must stay command-block-first, use touch/proximity conversion or an add-on/script-backed hit event.

Alternative Note: “so the whole map knows it happened” conflicts with the later command-logic line that limits the boom to players within range / 50 blocks. Decide whether the sound is map-wide or local-range and keep both sections consistent.

Builder implementation

  • Conversion moments must happen in readable spaces with enough openness for nearby witnesses and enough route continuity for chain pursuit.
  • Avoid enclosed micro-rooms where conversion FX are visually wasted.

Developer implementation

  • Conversion transaction must be atomic: remove survivor state, grant bearer state, clear survivor kit, grant bearer kit, apply FX, increment counters, evaluate survivor count, and check end conditions.
  • Flash can be implemented more faithfully with the /camera system and fade options if a true screen flash is required; an actionbar message alone is not equivalent to a white flash. The documented camera command supports fades, fixed positions, and facing controls.

3.2 Survivor Tools

Survivor toolkit

“Survivors have limited tools to slow Bearers and buy time — they cannot eliminate them:

  • 3x Snowballs on spawn — slow a Bearer for 1 second on hit
  • Speed I boots — allows evasive movement
  • A faint gold glow on all uncorrupted players so survivors can see each other”

Alternative Note: A gold-colored Glowing outline is not established by the cited Bedrock command documentation. If color-specific glow cannot be verified in the target runtime, replace this with a gold particle marker, armor accent, or below-name/readability solution while preserving the original intent of survivor visibility. Supported scoreboard display slots are sidebar, list, and belowName.

Builder implementation

  • Survivors require interconnected evasive routes, corner breaks, vertical transitions, and soft cover.
  • Routes must reward quick tactical decisions, not long-term camping.

Developer implementation

  • Snowball slow can be implemented with command-block projectile proximity logic similar to Microsoft’s documented snowball-fight example, then applying a brief slowness effect to Bearers near the projectile contact point.
  • Speed I boots should be granted at round start to Survivors and removed on conversion.
  • “Cannot eliminate Bearers” means no damage race condition should ever allow Survivors to kill or remove Bearers from active play.

Intended power balance

“Survivors cannot eliminate Bearers. The power dynamic should always favour the Bearers as the round progresses. This is intentional design.”

Builder implementation

  • Do not create defensive architecture that lets one survivor lock a win through geometry alone.

Developer implementation

  • All slowdown tools must be temporary and non-lethal.
  • Bearer scaling can intensify indirectly as the survivor population drops.

3.3 Win Condition

Last-survivor win sequence

“The last uncorrupted Survivor triggers the win sequence:

  • /title fires: ‘LAST SURVIVOR — [PlayerName]!’
  • Fireworks launch from the Crystal pedestal
  • A victory sound plays across the map
  • Scoreboard updates with a round win for that player
  • 10-second countdown, then map auto-resets — no teacher action required”

Builder implementation

  • Hub and pedestal must be visually framed for fireworks and round-end spectacle.

Developer implementation

  • On survivor count = 1, immediately lock winner, prevent duplicate win firing, increment round win, launch ceremony, then enter reset countdown.
  • Fireworks can be staged from pedestal coordinates via summon chains as specified later.

Timer-expiry resolution

“If the timer expires before a last survivor is identified, the final remaining uncorrupted player is declared the winner. If multiple survivors remain at expiry, the round is a draw and no point is awarded.”

Developer implementation

  • Timer-expiry branch must count survivors once, then resolve:
    • 1 survivor → win
    • 2+ survivors → draw
  • Draws must not increment streaks or wins.

4. Map Layout & Structure

Map footprint

“The map is an ancient ruined town… Recommended footprint: 100 x 100 blocks.”

Builder implementation

  • Use 100x100 as the production envelope for live arena space, excluding hidden redstone/add-on maintenance cavities if necessary.
  • Keep the center of mass near Town Square so circulation remains intuitive.
  • The map must read as a single cohesive town rather than disconnected encounter rooms.

Developer implementation

  • Spawn, spectator camera, countdown gate, and ceremony anchors must be positioned relative to this footprint for predictable selector ranges and FX coverage.

4.1 Zone Overview

Spawn Hub

“Shared pre-round room. Crystal pedestal, leaderboard wall, ready board, exit gate.”

Builder implementation

  • Must be physically separated from the live combat space while still emotionally connected to the arena.
  • Use a strong front-facing composition with pedestal center, walls flanking, exit gate as clear round launch vector.

Developer implementation

  • This zone owns all pre-round logic, ready checks, round dispatch, post-round returns, and session-end display.

Town Square

“Open stone plaza — 20x20 blocks.”

Builder implementation

  • This is the main confrontation bowl.
  • Must allow sightlines from multiple street entries and at least partial visibility from rooftops.
  • The DAIGON shield floor inlay belongs here and must remain readable from above.

Developer implementation

  • Suitable for titles, visible FX, and final-two tension.
  • Avoid heavy entity logic anchored here unless performance-tested.

Street Network

“4 intersecting streets, 3 blocks wide.”

Builder implementation

  • Three-block width is the minimum clear travel width.
  • Decorative projections must not reduce the functional chase lane below intended width.
  • Every street must visually telegraph its destination zone.

Developer implementation

  • Use streets as route logic anchors for coverage testing and bearer path timing.

Building District

“3 multi-floor buildings (2–3 floors).”

Builder implementation

  • Buildings must be competitively legible, not dense roleplay interiors.
  • Internal staircases must be short, readable, and safe to navigate under pressure.
  • Window openings and balcony lines should support visual awareness without enabling abusive unreachable positions.

Developer implementation

  • Interior block complexity must not interfere with command selectors if area-based logic is later used.

Rooftop Layer

“Connected rooftop paths across the building district. Rope bridges between buildings.”

Builder implementation

  • Rooftops must create a risk-reward layer, not an untouchable safe zone.
  • Bridges need readable approaches, landings, and escape consequences.

Developer implementation

  • If rooftops are intended for survivor advantage, Bearer mobility must still allow meaningful contest.

Side Alleys

“Two narrow alleys (1–2 blocks wide) behind the buildings.”

Builder implementation

  • Narrow means tense, not obstructive.
  • Alleys need light, route continuation, and at least one decision point each.

4.2 Design Rules

No dead ends

“No dead ends — every corridor must have at least two exits”

Builder implementation

  • This is a hard rule.
  • Any storage niche, balcony nook, stair landing, or alley recess must either reconnect or be visibly too shallow to be mistaken for an escape route.

No completely dark areas

“No completely dark areas”

Builder implementation

  • Maintain warm, readable ambient lighting throughout.
  • Use hidden light blocks or embedded light sources if needed, but preserve authored lantern/crystal look.

Destructible elements

“Destructible elements — select fences and wooden doors breakable by the Bearer to prevent camping”

Alternative Note: Bearer-only destruction needs an explicit technical method. In shared survival/adventure rules, this is not automatic. Either give Bearers a controlled can-destroy tool/item path or replace breakable elements with scripted removals that only trigger for Bearers.

Builder implementation

  • Destructible pieces must be sparse, obvious, and placed only where anti-camping value is high.
  • Never make core navigation depend on a destructible object.

Developer implementation

  • Destruction targets must reset automatically at round end.
  • Do not allow permanent world damage across rounds.

Rooftop access

“Rooftop access — at least two staircase routes to rooftops, one per side of the building district”

Builder implementation

  • Stair entry points must be discoverable within seconds.
  • Each access route must remain viable even during player congestion.

Crystal veins

“Crystal veins — decorative purple crystal veins… visual only”

Builder implementation

  • Veins should behave as environmental storytelling paths from hub origin to arena fabric.
  • They must not be mistaken for interactable mechanics.

4.3 Visual Identity & Atmosphere

Builder implementation

  • Palette: sandstone, oak, mossy cobblestone, blackstone/obsidian only where crystal corruption or branded premium treatment is intended.
  • Lighting: warm base light with selective purple crystal contrast.
  • Branding: visible but integrated into architecture and streetscape.
  • Mood: adventurous, magical, competitive, never horror.

5. Spawn Hub

Hub overview

“Before every round, all players load into the shared Spawn Hub. This is where the round starts, the Bearer is announced, and students check the leaderboard and ready up between rounds.”

Builder implementation

  • Hub must support standing groups of 16 with clear view to leaderboard and ready board.
  • Reading walls must not compete with the exit arch for focus.

Developer implementation

  • All round transitions must resolve here.
  • Teleport endpoints must be distributed enough to avoid body stacking.

5.1 Layout

Central open room

“Central open room — minimum 20x12 blocks”

  • Builders: treat as minimum clear footprint, excluding wall thickness and decorative recesses.

Dark Crystal pedestal centrepiece

“Dark Crystal pedestal centrepiece”

  • Builders: pedestal must dominate the central axis.
  • Developers: coordinate anchor for fireworks and ceremony.

Leaderboard Wall

“Leaderboard Wall — minimum 16 blocks wide”

  • Builders: reserve large, uncluttered wall plane with premium framing.
  • Developers: leave hidden access cavity if scripting/sign refresh/support blocks are needed.

Ready Wall

“Ready Wall — adjacent to the leaderboard. Displays each player's name with a green (ready) or amber (waiting) indicator updated in real time via command block.”

Alternative Note: A live 8–16 player named wall display is straightforward as a scoreboard/list UI, but a true in-world per-player wall with changing names and color states is much safer with Script API or a constrained display strategy. The current creator docs document scoreboards and sign text scripting, but not a dedicated command for arbitrary dynamic sign text updates.

Rules board

“Rules board — sign board explaining the mechanic in 5 bullet points. Simple enough for an 8-year-old.”

  • Builders: place at eye level in low-noise location.
  • Developers: static content, no dynamic logic required.

Exit gate

“Exit gate — barrier blocks that open when all players are ready or teacher overrides. Seal again at round end.”

  • Builders: gate must read as a ceremonial release point.
  • Developers: gate state must be driven only by ready/override and round-state logic.

DAIGON ESPORTS banner above the exit arch

“DAIGON ESPORTS banner above the exit arch”

  • Builders: this is a hero branding placement.

5.2 Leaderboard Wall

Leaderboard wall display

“The leaderboard wall displays live stats via command block scoreboard output:

  • Columns: Player Name | Rounds Won | Survival Streak
  • Updated in real-time via sidebar scoreboard mirrored onto the wall using sign commands
  • DAIGON ESPORTS logo in pixel-art banner form above the leaderboard
  • Framed in polished blackstone with purple-tinted glass accents”

Alternative Note: Sidebar scoreboards are officially supported, but “mirrored onto the wall using sign commands” is a high-risk phrasing unless scripting is allowed. Script API documents sign text mutation via BlockSignComponent.setText, while the public command references do not document a general sign-text command equivalent. Prefer one of these locked options: 1) sidebar only, 2) sidebar + scripted wall sync, or 3) a static branded wall plus live UI scoreboard.

Builder implementation

  • The wall is a presentation object and must be readable from normal standing distance.
  • Frame depth, contrast, and spacing must prioritize legibility over ornament density.

Developer implementation

  • rounds_won and streak must exist as persistent objectives.
  • If a world-space wall is required, reserve scripting scope for sign updates or equivalent dynamic display.

Platform handoff baseline

“DAIGON's scripting team will connect the scoreboard data to the DAIGON platform post-delivery… MIVUBI delivers a fully functional command block scoreboard as the baseline.”

Developer implementation

  • Deliver a clean baseline objective schema with stable names and deterministic update moments.
  • Do not hardcode assumptions about external platform ingestion inside map logic.

6. Game Flow

Required flow

Canonical flow
Spawn Hub
→ Players Ready Up
→ Bearer Selected
→ Gate Opens
→ Survivors Scatter
→ Conversions
→ Last Survivor Wins
→ Auto-Reset
→ Next Round

Developer implementation

This should be treated as the canonical state order.

Spawn Hub phase

  • Players gather
  • Check leaderboard
  • Click ready NPC
  • Ready Wall updates
  • Round blocked until all ready or teacher override

Developer implementation

  • Use NPC-triggered tagging or a ready objective.
  • @initiator is documented as a selector for NPC dialogue interactions.

Bearer Selection phase

  • Random player selected
  • /title to bearer and others
  • Sound plays
  • Kit/effects applied

Developer implementation

  • The bearer announcement chain must not open the gate early.
  • All role effects must be active before countdown ends.

Countdown phase

  • 5 seconds
  • animated countdown
  • gate opens at zero
  • bell sound

Builder implementation

  • Gate reveal and first sightline out of hub should feel dramatic.
  • Immediate branching routes must be visible after gate release.

Combat phase

  • Bearer hunts
  • conversions occur
  • conversion count on sidebar

Developer implementation

  • Sidebar must prioritize the most useful live metric during combat.
  • Avoid constantly changing too many objectives mid-round.

Last Survivor / Auto-Reset / Session End

  • Must be fully automated and ceremony-driven.
  • Session-end winner announcement must freeze final result state before reset/despawn cleanup.

7. Fun & Engagement Features

Corruption Aura

  • Builders: keep aura readable against environment.
  • Developers: escalating intensity can be tied to bearer-count score and layered particle commands. Particle systems are supported in creator docs.

Conversion Flash

Alternative Note: A “brief white flash” is not equivalent to an actionbar message. If the white-flash effect is required, use the documented camera/fade system or another full-screen visual method instead of only text.

Last Two Alert

  • Builders: final-two moments should be strongest in readable center routes and rooftops.
  • Developers: fire only once per round when survivor count transitions from 3 to 2.

Bearer Conversion Count

  • Developers: per-player round-scoped objective; reset at round start.

Survival Streak Shoutout

  • Developers: the brief defines the streak as consecutive rounds survived/won; use the same definition consistently in session UI and telemetry.

Spectator Mode

  • Builders: define one or more overhead camera/view positions with strong whole-map readability.
  • Developers: spectator mode is supported in current Bedrock docs, and spectators cannot use items or interact while spectating. If the compass is meant to enter spectator mode, the activation must happen before the mode switch, or the feature should use /camera instead.

Victory Firework Show

  • Builders: leave vertical clearance above pedestal.
  • Developers: stagger launches over fixed 10-second session-end window.

8. DAIGON Branding

All elements in Section 8 are mandatory build requirements, not optional dressing.

Spawn Hub — above leaderboard

10. Data Capture

Telemetry baseline

“All data capture is handled via the Minecraft server data pipeline… MIVUBI must ensure all scoreboard objectives and event triggers listed below are implemented and firing correctly.”

Alternative Note: Scoreboards are documented for numeric tracking and display, but the event tables here include string payloads such as player_id, session_id, school_id, map_id, and timestamp-style metadata. Those are not ordinary scoreboard values. Deliverable scope should distinguish between 1) numeric in-world objectives, 2) deterministic event moments, and 3) external pipeline enrichment that attaches non-numeric metadata.

Developer implementation

  • Deliver an objective schema for numeric counters.
  • Deliver a stable set of trigger moments for external ingestion.
  • Do not promise that vanilla scoreboards alone serialize all listed non-numeric fields.

10.1 Player-Level Data

For every listed field, implement one of the following classifications:

  • Numeric objective in-world: rounds played, rounds won, times as bearer, total conversions, streak, snowballs used, times converted
  • Derived metric external or scripted: win rate, average conversions, average survival time, best survival time
  • Session metadata external/pipeline-owned: player ID, session ID, school ID, session timestamp

Developer implementation notes

  • Survival time requires precise round-start and conversion/win timestamps or tick counters.
  • “Rounds as last survivor” must be incremented only on authentic solo survival wins, not draws or expiry ties.
  • Current survival streak and longest survival streak must be tracked separately.

10.2 Club-Level Data

Developer implementation

  • These are aggregation outputs, not all necessarily native in-world displays.
  • The map must emit consistent per-session counts and per-player counters so the external platform can aggregate club metrics.

  6. Education deployment test — playable export in Minecraft Education Edition

12. Consolidated Inline Risk Register

This section does not replace the inline notes above; it summarizes them for production planning.

  1. Conversion trigger conflict: touch vs sword-hit/damage
  2. Visible BEARER tag conflict: /tag is not a visible overhead label
  3. Gold glow requirement: color-specific glowing is not verified in cited command docs
  4. Ready Wall / Leaderboard Wall: true live in-world named displays are safer with scripting than command-only sign assumptions
  5. Hit detection: high risk if command-block-only
  6. Boom scope conflict: map-wide vs 50-block radius
  7. White flash requirement: actionbar text is not a true screen flash
  8. Telemetry payload mismatch: scoreboards do not natively represent all non-numeric metadata fields
  9. Late join/session semantics: SESSION_START behavior needs session-rule clarification

13. Recommended Production Lock

To preserve the brief’s intent while reducing implementation risk, production should lock these decisions before build/code freeze:

  • Authoritative conversion trigger
  • Visible Bearer labeling method
  • Survivor-visibility method
  • Whether live wall displays require scripting
  • Whether spectator uses /gamemode spectator, /camera, or both
  • Exact telemetry ownership split between in-world objectives and external pipeline enrichment

These decisions do not change the design intent. They only prevent late-stage implementation ambiguity.
